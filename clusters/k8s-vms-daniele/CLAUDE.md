# k8s-vms-daniele — App-Specific Gotchas

Terse pointers to release.yml decisions that used to be inline comments — full rationale and any incident detail now lives in Confluence.

- `awx` — the ingress host is Cloudflare-Tunnel-fronted, not exposed via this Ingress's own LB status; requires explicit `external-dns.alpha.kubernetes.io/target` annotation (tunnel target from cluster-vars) to opt in correctly. See [Confluence: AWX](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/748584961).
- `cloudnative-pg-operator` — this is a different operator from kubenuc's Zalando postgres-operator convention (different CRDs entirely); `podMonitorEnabled` is left off since Prometheus Operator CRDs aren't confirmed present on this cluster. See [Confluence: CloudNativePG Operator](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/777551916).
- `external-dns` — CF_API_TOKEN needs Reloader wiring to restart on secret rotation; `txtOwnerId` is permanent, changing it orphans every owned record; adoption is opt-in via annotation only. See [Confluence: External DNS](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/774012929).
- `grafana-alloy` — required secret-key contract on `grafana-cloud-credentials`; collector capabilities must stay at chart defaults or the logs collector breaks; log-severity drop/relabel pipeline keeps active-series budget in check. See [Confluence: Grafana Alloy](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/748257281).
- `mcp-viewer` — bound SA token minted via `kubectl create token` defaults to a 2h TTL and expires silently, still requiring periodic regeneration; the MCP kubeconfig now reaches this cluster over a direct VPN address instead of Teleport's kube-access proxy, because Teleport separately enforces its own `kubernetes_users`/`kubernetes_groups` impersonation allow-list independent of K8s RBAC — a `kubectl auth can-i --as=` test run through Teleport can fail (or pass) without saying anything about the SA's real RBAC. See [Confluence: mcp-viewer](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/777814017) for both failure modes, the VPN-direct fix, and the corrected verification command.
- `reloader` — needs the dedicated `readOnlyRootFileSystem` top-level flag (not `containerSecurityContext`) or the required `/tmp` mount is never added; UID pinned to house style (65532) instead of the chart's own default. See [Confluence: Reloader](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/777551893).
- `semaphore` — `admin.*` (not `auth.admin.*`, this chart has no `auth` key); a later 1Password `admin-password` change does NOT propagate to an existing user — use `semaphore user change-by-login` inside the pod instead; `securityContext`/`podSecurityContext` render at the opposite levels from the usual chart convention. See [Confluence: Semaphore](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/748552193).
- `semaphore` chart version is now an exact pin (`16.2.2`), not an open range — an open range gives Renovate nothing to bump, so the chart was silently re-resolving to newest-matching on every reconcile with no git commit at all. Every Renovate PR that bumps this pin is now also the trigger to rebuild/push `docker/semaphore-runner` (see `runner.yml` below) — the two images track different upstreams (`semaphoreui/semaphore` vs `semaphoreui/runner`) and don't move in lockstep automatically.
- `semaphore-runner` (`manifests/runner.yml`) — dedicated remote runner, not a fork of the server's own image; moves task execution off the server pod so the server chart itself never needs sops/1Password tooling forked into it. The runner's `web_host` (which server to register with) is config-file-only — no `SEMAPHORE_*` env var exists for it, confirmed against the runner binary and Semaphore's own docs — hence the small `semaphore-runner-config` ConfigMap rather than pure env vars. The `semaphore-runner-data` PVC is required, not optional: it persists the registered runner's identity token across pod restarts; `runner-entrypoint.sh` only calls `runner register` when that persisted token is missing, since the registration token's single-use-vs-reusable behavior against the live server is unconfirmed. The runner's SOPS age key mount means the `semaphore` namespace can decrypt every `ansible/*/host_vars/*.sops.yaml` file, not just the ones a given task touches — full blast radius, not scoped per-task. The pod spec also needs `enableServiceLinks: false` — otherwise Kubernetes injects a legacy `SEMAPHORE_PORT=tcp://<clusterIP>:3000` env var from the server's own Service, and the runner CLI's config loader panics on it as an invalid port. It also needs `SEMAPHORE_RUNNER_PRIVATE_KEY_FILE` set (writable PVC path, e.g. `/data/runner_key.pem` — `/etc/semaphore` is read-only), or registration silently submits an empty public key and every job-poll fails with `"invalid public key"`. Any runner identity registered before that env var was set is permanently broken server-side and needs `semaphore runner unregister` plus deleting the persisted token file to force a clean re-registration. See [Confluence: Semaphore](https://fastnetserv.atlassian.net/wiki/spaces/IT/pages/748552193).
